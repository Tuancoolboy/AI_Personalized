1. Quyết định AI tự ra (ngoài spec)

# Quyết định Lý do
1 Tạo lib/supabase/employee.ts làm data access layer client-side cho employee flow. Giữ logic Supabase tập trung, tránh rải query trực tiếp vào nhiều component.
2 Dùng browser Supabase client + session user id cho profile/progress/quiz/time logs. Đúng RLS user-scoped, không cần service role và không expose secret ra client.
3 Vẫn ghi local demo profile sau khi onboarding Supabase lưu thành công. Giữ trải nghiệm hiện tại ổn trong cùng browser và không ảnh hưởng DB thật.

2. Những chỗ AI thay đổi so với spec gốc

# Spec nói AI làm gì thay thế Lý do
1 Fallback localStorage cho demo mode. Fallback chỉ là nhánh chính khi thiếu Supabase env; khi Supabase write fail thì hiển thị lỗi và rollback/preserve safe state. Tránh che lỗi persistence thật, giúp debug Supabase dễ hơn.
2 Persist toàn bộ employee flow. AI tutor vẫn dùng canned responses, nhưng profile role cho tutor đã đọc từ Supabase real mode. OpenAI route nằm ngoài scope BE-07, nhưng tutor cần profile thật để không kẹt onboarding sau reload.

3. Tradeoff AI phải cân nhắc

Client data layer vs Route Handlers: Chọn client data layer vì các bảng đã có RLS user-scoped và component hiện là client components. Route Handlers sẽ thêm nhiều surface auth/error mới mà chưa cần cho Phase 1.

Upsert profile vs update-only: Chọn upsert kèm policy profiles_insert_own để xử lý cả user cũ chưa có profile do trigger chưa từng chạy.

Manager DB now vs later: Giữ manager mock theo AGENTS/BE-07 để tránh mở Phase 2 khi mục tiêu hiện tại là employee persistence.
